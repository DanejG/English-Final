Writing that lasts

Clean code is code that reads like a clear explanation.

This page is a practical guide to writing code that teammates can understand quickly, change safely, and trust over time.

Quick rules

If you only remember 6 things.

  1. Name things so you rarely need comments.
  2. Keep functions small and single-purpose.
  3. Prefer clear flow over clever tricks.
  4. Reduce duplication: one idea, one place.
  5. Handle errors early and explicitly.
  6. Write tests for behavior, not implementation.

Principles that scale

Clean code is less about style and more about reducing reader effort. These principles work in any language.

Make intent obvious

Readers should understand why something exists at a glance. Use specific names, small functions, and clear data shapes.

  • Prefer isExpired over flag
  • Make units explicit: timeoutMs
  • Let types/structures tell the story

Keep complexity local

Complex logic is unavoidable sometimes. Keep it boxed-in, well named, and tested so it doesn’t leak.

  • Extract helpers with meaningful names
  • Prefer pure functions for tricky logic
  • Minimize hidden shared state

Optimize for change

Most time is spent modifying existing code. Clean code makes change low-risk by being easy to reason about.

  • Reduce duplication
  • Design small interfaces
  • Keep “setup” separate from “work”

Errors are part of the API

Defensive code clarifies what can go wrong. Handle failures early, close to the source, with useful messages.

  • Validate inputs at boundaries
  • Return/throw explicit errors
  • Prefer guard clauses

Example 1: naming + guard clauses

function f(u) {
  if (!u) return false;
  if (u.a && u.b && u.c) {
    if (u.t && Date.now() - u.t < 86400000) return true;
  }
  return false;
}
const ONE_DAY_MS = 24 * 60 * 60 * 1000;

function hasValidSession(user) {
  if (!user) return false;
  if (!user.account || !user.profile || !user.permissions) return false;
  if (!user.lastLoginAt) return false;

  const sessionAgeMs = Date.now() - user.lastLoginAt;
  return sessionAgeMs < ONE_DAY_MS;
}

Cleaner code often adds a few lines but removes confusion: clearer names, explicit units, and early exits.

Example 2: separate “choose” from “do”

function ship(order, customer) {
  let cost = 0;
  if (customer.vip) cost = 0;
  else if (order.total > 50) cost = 0;
  else if (order.total > 25) cost = 4.99;
  else cost = 7.99;

  send(order, cost);
}
function shippingCostFor({ total }, { isVip }) {
  if (isVip) return 0;
  if (total > 50) return 0;
  if (total > 25) return 4.99;
  return 7.99;
}

function ship(order, customer) {
  const cost = shippingCostFor(order, { isVip: customer.vip });
  send(order, cost);
}

By extracting the decision into a pure function, you can test it easily and keep the “shipping” action simple.
